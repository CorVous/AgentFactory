# Concurrency cap (`--max-concurrent-foremen`)

## Parent

[PRD-0001: Ralph-Loop mesh for AgentFactory V1](../PRD.md)

## What to build

Wire the Kanban's hardcoded max-concurrent-Foremen value (placeholder from #03) to a configurable launcher flag, so a flood of ready issues cannot melt the user's machine or budget (story #6).

- Launcher accepts `--max-concurrent-foremen <N>` and forwards it to the Kanban as a CLI flag.
- Kanban's spawn-decision pure function already takes `maxConcurrent` as a parameter (per #03's shape). This issue adds the flag plumbing and a sensible default (e.g., `2` or `4` — author's call, document in the launcher help).
- The pure function's existing unit tests cover the cap behaviour; this issue adds tests for the flag-default and flag-override paths through the launcher.

## Acceptance criteria

- [ ] Launcher accepts `--max-concurrent-foremen <N>`; missing flag uses a documented default.
- [ ] Kanban receives the value via the existing CLI-flag plumbing from `scripts/run-agent.mjs` (or the Kanban-specific launcher equivalent).
- [ ] Spawn-decision pure-function unit tests cover: `N=1` allows one Foreman in flight, `N=3` allows three, ready-but-over-cap issues are deferred to the next wake.
- [ ] Manual verification: a fixture project with N+1 ready issues sees exactly N Foremen alive concurrently; the (N+1)th dispatches when one of the first N exits.

## Agent Brief

**Category:** enhancement
**Summary:** Wire the `--max-concurrent-foremen <N>` flag through the launcher to the Kanban's spawn-decision pure function (which already accepts `maxConcurrent` after #03).

**Current behavior:**
After #03 lands, the Kanban hardcodes its max-concurrency value and ignores any launcher flag. A flood of ready-for-agent issues would dispatch as many Foremen as it can, with no user-configurable cap.

**Desired behavior:**
The launcher accepts `--max-concurrent-foremen <N>`; missing flag uses a documented default (suggest `2`, but author's call). The value is forwarded to the Kanban via the existing CLI-flag plumbing. The Kanban's spawn-decision pure function honours it: `N=1` allows one Foreman in flight, `N=3` allows three, ready-but-over-cap issues defer to the next wake.

**Key interfaces:**
- Launcher CLI flag — `--max-concurrent-foremen <N>` with documented default in `--help`.
- Existing CLI-flag plumbing (post-#03) that already passes other flags from the launcher to the Kanban.
- Spawn-decision pure-function signature `(issueTreeState, currentForemen, maxConcurrent) → spawnDecisions[]` — the parameter exists; this issue adds tests for varied `maxConcurrent` values.

**Acceptance criteria:**
See the issue body's Acceptance criteria section.

- Per-issue priority overrides.
- Rate-limiting or token-budget caps (concurrency only).
- Dynamically adjusting the cap mid-run.
